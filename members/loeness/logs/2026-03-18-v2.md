# Log: Quantitative RAM Analysis for Full-Stack AegisLab Development

## 1. Problems
When running a multi-node Kubernetes cluster (Kind) along with infrastructure dependencies (Chaos Mesh + Prometheus Stack), memory pressure often leads to **OOM (Out Of Memory) kills** or extreme system latency due to disk swapping. 

**Theoretical Memory Consumption Breakdown:**
| Component Layer | Runtime Processes | Estimated RAM Usage |
| :--- | :--- | :--- |
| **Linux OS & Docker** | Kernel, Docker Daemon, Systemd | **1.5 - 2.0 GB** |
| **Kind Control Plane** | API Server, ETCD, Controller Manager | **1.5 - 2.5 GB** |
| **Kind Worker Nodes (x2)** | Kubelet, Containerd, Proxy | **1.0 - 2.0 GB** |
| **Monitoring Stack** | **Prometheus**, Grafana, OTEL Collector | **2.5 - 5.0 GB** |
| **Chaos Mesh** | Dashboard, Controller, Daemons | **0.5 - 1.0 GB** |
| **AegisLab Services** | Producer, Consumer, MySQL, Redis | **1.0 - 2.0 GB** |
| **TOTAL (Idle)** | **Baseline Requirement** | **~8 GB - 14 GB** |

*Note: Prometheus is the primary "RAM hog," as it keeps time-series indices in memory.*

---

## 2. What I Did
To accurately assess and manage memory resources on a host machine, the following diagnostic and optimization steps were established:

* **Real-time Monitoring**: Utilized `htop` and `docker stats` to distinguish between host-level memory usage and container-specific limits.
* **Cgroup Limit Analysis**: Investigated the memory limits set within the Kind configuration files to prevent a single "rogue" pod (e.g., a stress-test consumer) from crashing the entire host OS.
* **Swap Management Check**: Verified the status of `swappiness` in the Linux kernel (`cat /proc/sys/vm/swappiness`) to ensure the system doesn't prematurely dump active memory to the slower disk.

---

## 3. Successful
A stable "Healthy" state for a host machine is defined by maintaining **at least 20% Headroom** under peak load (e.g., during a Chaos experiment).

**Ideal Resource Allocation:**
* **Host RAM**: 16 GB (Minimum), 32 GB (Recommended).
* **Swap Space**: At least 4 GB allocated on an NVMe SSD to handle transient spikes without freezing the UI.
* **Component Health**: All Kubernetes nodes report `MemoryPressure=False`.



---

## 4. Suggestion
* **The 16GB Baseline**: For developers using a standard laptop/PC, **16GB of physical RAM** is the absolute baseline. Attempting to run the full AegisLab stack (Kind + Chaos + Monitoring) on 8GB will likely result in system-wide instability.
* **Prometheus Tuning**: If memory is tight, suggest reducing the Prometheus `retention.time` or `storage.tsdb.max-block-duration` to limit the memory-mapped file index size.
* **Selective Deployment**: During early development phases, consider disabling the heavy Monitoring stack (Grafana/Prometheus) unless specifically testing observability logic, which can save ~4GB of RAM.
