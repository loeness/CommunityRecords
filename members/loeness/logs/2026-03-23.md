# Troubleshooting Report: K8s Control-Plane Failure

### What happened?
The Kind cluster creation failed at the **"Starting control-plane"** stage. The `kubelet` failed to initialize within the 4-minute timeout, causing the script to abort and automatically delete the failed nodes, leaving the environment unreachable.

### Why did it happen?
The system defaulted to **Kubernetes v1.35.0**, an unstable experimental version. This created a conflict with the project's legacy `v1beta3` API configuration. Additionally, incompatibility between the host’s **Cgroup v2** and the experimental `kubelet` caused the service to crash.

### What was done?
The corrupted state was cleared using `kind delete cluster`. The cluster was then manually re-provisioned by pinning a stable LTS image:
```bash
kind create cluster --name test --config manifests/test/kind-config.yaml --image kindest/node:v1.31.0
```
This successfully initialized the control-plane, CNI, and worker nodes.

### Advice for Backend Developers
* **Prioritize Stability**: Never rely on "latest" or default versions for infrastructure. Always pin container images to specific, stable LTS releases (e.g., `v1.31.x`) to ensure environment reproducibility.
* **Sync IaC Templates**: Regularly update your **Infrastructure as Code** (YAML) files to match the API specifications required by the target Kubernetes version.
* **Debug Inside Containers**: If `kubectl` fails, use `docker exec` to check `journalctl -u kubelet` inside the node container to identify the root cause of "connection refused" errors.
