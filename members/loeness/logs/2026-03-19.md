# Log: Chaos Mesh Installation Hangs due to TLS Handshake Timeout in Kind Environment

## 1. Problems
While executing the environment setup script `./scripts/start.sh test`, the process hangs indefinitely at the following stage:

```text
Installing Chaos Mesh...
"chaos-mesh" has been added to your repositories
Attempt 1/3...
```

**Phenomena:**
* Checking the cluster status with `kubectl get pods -n chaos-mesh` shows that the pods are stuck in `ImagePullBackOff` or `ErrImagePull`. 
* Detailed inspection via `kubectl describe` reveals a **TLS handshake timeout**.
* **Root Cause:** Kind nodes are unable to reach the external image registry (`pair-diag-cn-guangzhou.cr.volces.com`) from within the containerized network, likely due to internal network isolation or proxy issues.

---

## 2. What I Did
To resolve the network bottleneck within the cluster, I implemented an **Image Sideloading** workflow:

### Step 1: Diagnostic Verification
Confirmed the host machine could reach the registry by manually pulling one of the images:
```bash
docker pull pair-diag-cn-guangzhou.cr.volces.com/pair/chaos-mesh/chaos-mesh:v2.8.0
```

### Step 2: Sideloading to Cluster
Manually pushed the required infrastructure images from the host's Docker engine into the Kind nodes to bypass the internal image puller:
```bash
kind load docker-image pair-diag-cn-guangzhou.cr.volces.com/pair/chaos-mesh/chaos-mesh:v2.8.0 --name test
```

### Step 3: Manual Helm Deployment
Overrode the default Helm chart configurations to point to the local sideloaded images, ensuring 100% local resolution for the control plane.

---

---

## 4. Suggestion
* **Use Sideloading for Restricted Networks:** In environments with strict network isolation or unstable registry access, it is recommended to pre-pull images on the host and use `kind load` instead of relying on the cluster's internal pull mechanism.
* **Enhanced Monitoring:** If a setup script hangs without verbose output, developers should immediately open a secondary terminal to monitor `kubectl get pods -A` to identify underlying image pull failures early.
